# 《操作系统》课程知识图谱 — Schema 设计文档

> 版本 v2.0 · 用途:作为知识抽取的"靶子"与答辩时阐述设计的依据
> 原则:**每一类节点、每一条边都对应一个下游功能,不设计无用结构。**
>
> **v2.0 修订要点**:
> - 算法不单列为独立节点类型 —— 算法本质上就是 Concept,只是带额外可比较字段;改用 `node_role` 属性 + 专属可选字段表达。理由见 §1.1。
> - 新增 `SOLVES` 边,解锁"问题→机制/算法"的因果链问答(GraphRAG 多跳推理的优质素材)。

---

## 0. 设计总览

| 维度 | 内容 |
|---|---|
| 节点类型 | `Concept`(概念,含算法/机制/问题,用 `node_role` 区分)、`Problem`(习题) |
| 边类型 | `PREREQUISITE`、`PART_OF`、`RELATED`、`CONFUSABLE`、`TESTS`、`SOLVES` |
| 存储 | NetworkX（默认，内存图）/ Neo4j（可选懒加载）；概念 1389，边 4978，习题 306 |
| 核心命脉 | `PREREQUISITE`(支撑学习路径 + 认知诊断溯源) |

**关键设计决策:单一主体节点类型。** 算法、机制、问题都是 `Concept`,靠 `node_role` 属性区分,而非新建节点类型。这样它们天然继承 Concept 的全部能力(先修、被考查、解决问题),零额外逻辑,图谱也更简洁好维护。

## 1. 节点类型(Node Types)

### 1.1 Concept(概念)— 唯一主体节点

代表一个"教学知识点级别"的概念。粒度标准:能作为一道题考查、能在课件中独立成小节的,才算一个概念;过细的实现细节不单独建节点。

**为什么算法不单列为节点类型?** 算法(FCFS、LRU、银行家算法…)就是概念的一种:它一样有定义、有难度、有先修关系、会被习题考查、能解决某个问题。它没有任何 Concept 没有的"结构性行为",只是多了几个可比较字段(是否抢占、是否会饥饿等)。按"类型 vs 属性"的建模原则,**同类但带专属字段的情况应用属性表达,而非新增类型**。机制、问题同理。

| 属性 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `name` | string | 是 | 规范名称,全图唯一(如统一用"进程控制块"而非"PCB") |
| `definition` | string | 是 | 一句话定义(20–50字),供 RAG 上下文与诊断展示使用 |
| `chapter` | string | 是 | 所属章节(如"3.内存管理"),用于覆盖度自检与按章筛选 |
| `difficulty` | int (1–3) | 是 | 难度:1基础 / 2进阶 / 3综合。供路径规划排序与认知诊断加权 |
| `node_role` | enum | 是 | `概念` / `机制` / `算法` / `问题`,标记概念的角色,支撑分类查询与对比功能 |
| `aliases` | string[] | 否 | 别名列表(如 ["PCB"]),供 RAG 实体链接匹配,但不作为独立节点 |
| **以下为 `node_role="算法"` 时填写的可选专属字段** | | | |
| `preemptive` | bool | 否 | 是否抢占式 |
| `starvation_free` | bool | 否 | 是否避免饥饿 |
| `complexity` | string | 否 | 复杂度 / 调度开销描述 |
| `scenario` | string | 否 | 典型适用场景 |

> **算法对比功能怎么实现**:查询时筛 `node_role = "算法"` 的节点,读取它们的 `preemptive`/`starvation_free`/`complexity` 等字段,即可自动生成横向对比表。完全不需要独立节点类型。

#### node_role 取值说明

| node_role | 含义 | 示例 |
|---|---|---|
| `概念` | 一般知识点 | 进程、地址空间、临界区 |
| `机制` | 解决某问题的方法/手段 | 信号量、管程、分页机制 |
| `算法` | 具体算法(填专属字段) | 时间片轮转、LRU、银行家算法 |
| `问题` | 需要应对的现象/问题 | 死锁、内存碎片、优先级反转 |

### 1.2 Problem(习题)— 诊断锚点

代表一道习题,是认知诊断的"输入锚点":学生做错某题 → 经 `TESTS` 边定位到所考概念 → 经 `PREREQUISITE` 反向溯源到薄弱先修点。

| 属性 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `pid` | string | 是 | 习题唯一编号,格式 `CH{章号}_Q{题号}`,如 `CH3_Q2` |
| `stem` | string | 是 | 题干(保持原文) |
| `answer` | string | 否 | 参考答案;教材无答案时留空字符串,后续人工或 LLM 补充 |
| `type` | enum | 是 | 认知层次:`记忆` / `关系` / `综合`(判断标准见 §5) |
| `chapter` | string | 是 | 来源章节,格式与 `Concept.chapter` 一致,如 `"3.处理机调度与死锁"` |

## 2. 边类型(Relation Types)

> 表中"对应功能"说明这条边为什么存在 —— 删掉它,哪个功能就做不了。

### 2.1 PREREQUISITE(先修依赖)★命脉

- **方向**:有向。`A -[:PREREQUISITE]-> B` 读作「A 是 B 的先修」,即不理解 A 就无法理解 B。
- **约束**:全图在此边上必须是 **有向无环图(DAG)**。入库后必须跑环检测,有环即依赖标错,需人工修。
- **对应功能**:学习路径规划(拓扑排序)、认知诊断溯源(从错误知识点反向找薄弱先修点)。
- **建边策略**:LLM 生成候选 + **人工精校(必须)**。最易错也最关键的边。
- 示例:`地址空间 → 分页机制 → 虚拟内存`

### 2.2 PART_OF(从属/包含)

- **方向**:有向。`A -[:PART_OF]-> B` 读作「A 从属于 B / A 是 B 的一种或一部分」。
- **对应功能**:构建知识层级,支撑按主题聚合展示、子图召回的范围界定。
- **建边策略**:**半自动**。直接取自课件/教材的章节-小节层级结构,准确率最高,几乎不需 LLM。
- 示例:`时间片轮转 → 进程调度算法`、`分页 → 内存管理`

### 2.3 RELATED(关联)

- **方向**:无向。两概念话题相近、常一起出现,但**不构成先修关系**。
- **对应功能**:RAG 子图召回时扩展相关上下文;答辩时展示图谱连通性。
- **建边策略**:LLM 抽取,容错性高,无需重度人工核验。
- **注意**:务必和 `PREREQUISITE` 区分 —— "相关"不等于"先修",这是抽取时最常见的混淆。
- 示例:`信号量 — 管程`

### 2.4 CONFUSABLE(易混淆)

- **方向**:无向。连接一对经典易混概念。
- **属性**:`dimensions`(区分维度数组),每个维度记录 `{aspect 区分角度, a_value, b_value}`。
- **对应功能**:混淆诊断器 —— 用户描述理解 → 判断是否踩了某个典型误区 → 给出对比。
- **建边策略**:**人工为主**。经典混淆对数量有限且固定,人工列最准;LLM 仅用于补充区分维度。
- 示例:`进程 — 线程`、`并发 — 并行`、`死锁 — 饥饿`、`分页 — 分段`、`互斥 — 同步`

### 2.5 TESTS(习题考查)

- **方向**:有向。`Problem -[:TESTS]-> Concept`,表示该习题考查这些概念。
- **对应功能**:连接习题与概念,使"做错题 → 定位知识点 → 诊断溯源"链路成立。
- **建边策略**:LLM 抽取(给定概念白名单,让它在白名单内选),可信度较高。

### 2.6 SOLVES(解决/应对)★新增

- **方向**:有向。`A -[:SOLVES]-> B` 读作「A(机制/算法)解决 B(问题)」。
- **对应功能**:解锁"问题↔解决方案"的因果链问答(如"为什么需要信号量?"→ 它解决临界区问题)。这类"为什么"问题正是 GraphRAG 多跳推理的优质素材。
- **建边策略**:LLM 抽取 + 人工抽查。结合 `node_role`,通常从 `机制`/`算法` 指向 `问题`。
- 示例:`信号量 → 临界区问题`、`分页机制 → 内存碎片`、`银行家算法 → 死锁`

## 3. 边类型速查表

| 边 | 方向 | 端点 | 对应下游功能 | 建边策略 |
|---|---|---|---|---|
| `PREREQUISITE` | 有向(DAG) | Concept→Concept | 学习路径 + 诊断溯源 | LLM候选 + **人工精校** |
| `PART_OF` | 有向 | Concept→Concept | 知识层级 / 召回范围 | 半自动(取自章节结构) |
| `RELATED` | 无向 | Concept—Concept | RAG 上下文扩展 | LLM 抽取(容错高) |
| `CONFUSABLE` | 无向 | Concept—Concept | 混淆诊断器 | **人工为主** + LLM补维度 |
| `TESTS` | 有向 | Problem→Concept | 错题→知识点链路 | LLM(白名单内选) |
| `SOLVES` | 有向 | Concept→Concept | 因果链问答(多跳) | LLM 抽取 + 人工抽查 |

## 4. 完整示例片段

```
// 先修链:基础概念 → 机制 → 进阶概念
(地址空间:Concept {node_role:"概念", difficulty:1, chapter:"3.内存管理"})
   -[:PREREQUISITE]-> (分页机制:Concept {node_role:"机制", difficulty:2})
   -[:PREREQUISITE]-> (虚拟内存:Concept {node_role:"概念", difficulty:3})

// 层级:机制从属于主题
(分页机制:Concept) -[:PART_OF]-> (内存管理:Concept {node_role:"概念"})

// 解决关系:机制解决问题
(分页机制:Concept) -[:SOLVES]-> (内存碎片:Concept {node_role:"问题"})

// 算法节点:带专属可比较字段
(时间片轮转:Concept {
    node_role:"算法", difficulty:2, chapter:"2.进程调度",
    preemptive:true, starvation_free:true, complexity:"O(1)调度决策"
})
(银行家算法:Concept {node_role:"算法"}) -[:SOLVES]-> (死锁:Concept {node_role:"问题"})

// 易混淆:带区分维度
(分页:Concept) -[:CONFUSABLE {dimensions:[
     {aspect:"划分依据", a_value:"固定大小的页", b_value:"逻辑意义的段"},
     {aspect:"碎片类型", a_value:"内部碎片", b_value:"外部碎片"}
]}]- (分段:Concept)

// 习题连接概念
(P001:Problem {type:"综合", stem:"用银行家算法判断系统是否处于安全状态…"})
   -[:TESTS]-> (银行家算法:Concept)
   -[:TESTS]-> (死锁避免:Concept)
```

## 5. 习题认知层次(type 字段对照)

| type | Bloom 层次 | 对应图谱结构 | 判断标准 | 题目示例 |
|---|---|---|---|---|
| `记忆` | 记忆/理解 | 单个 Concept 节点 | 只需复述定义或列举特征,一句话能答 | "什么是信号量?" |
| `关系` | 应用/分析 | 两节点 + 一条边 | 需比较两个概念,或解释"为什么/有什么区别" | "信号量和管程的区别?" |
| `综合` | 综合/评价 | 子图 / 多跳 | 给具体场景让分析/计算,需要多个概念联合推理 | "用银行家算法判断以下系统是否安全" |

> 这张对照表也是"自动出题"功能的设计依据:按图谱结构的复杂度自动生成不同认知层次的题。

## 6. 入库后质量自检清单(必做)

- [ ] **环检测**:`PREREQUISITE` 是否成环?跑拓扑排序,报错即修。(最重要的一致性检查)
- [ ] **孤立节点**:是否有无任何边的 Concept?补边或删除。
- [ ] **覆盖度**:对照教材目录,核心章节有无遗漏。
- [ ] **实体唯一性**:有无别名分裂(同一概念被建成两个节点)?按 `aliases` 归一化。
- [ ] **边方向合理性**:抽查若干 `PREREQUISITE` 与 `SOLVES`,方向是否颠倒。
- [ ] **node_role 一致性**:`SOLVES` 的目标端点 `node_role` 是否多为"问题";算法节点是否填了专属字段。
